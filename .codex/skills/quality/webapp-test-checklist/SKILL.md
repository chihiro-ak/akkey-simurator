# webapp-test-checklist

## 目的
Webアプリに対して、  
**主要導線、入力、表示、レスポンシブ、エラー、外部連携** の観点からテストチェックリストを整理する。

## 使う場面
- 実装後の確認
- スプリントレビュー前
- リリース前
- 大きな画面変更後
- バグ修正後の回帰確認

## 入力
- `docs/14_ACCEPTANCE_CRITERIA.md`
- `docs/21_SCREEN_FLOW.md`
- `docs/22_UI_SPEC.md`
- 実装済み範囲の情報
- 既知不具合
- 必要に応じて `docs/50_TEST_PLAN.md`

## 出力
以下を含むテストチェックリストを出力する。

1. テスト対象範囲
2. 主要確認観点
3. ケース一覧
4. 優先度
5. 回帰確認項目
6. 未確認リスク

## テスト観点

### 1. 主要導線確認
- 開始から完了まで操作できる
- 進行不能がない
- 想定どおりの画面や状態に到達する

### 2. 入力・検証確認
- 必須項目が正しく検証される
- 不正値に対して適切にエラーが出る
- 保存、出力、送信結果が正しい

### 3. 表示確認
- レイアウト崩れがない
- 主要文言が分かりやすい
- 画像や参照ファイルの欠落がない
- 状態変化が視覚的に分かる

### 4. レスポンシブ確認
- デスクトップで主要操作が可能
- モバイルで主要操作が可能
- 横スクロールやタップ困難がない

### 5. エラー・外部連携確認
- 通信失敗時の表示が適切
- 入力不足時に詰まらない
- 外部連携失敗時の復帰手段がある

### 6. 体験確認
- 初見ユーザーでも迷いにくい
- 操作に対する反応が分かりやすい
- 致命的なパフォーマンス劣化がない

## 出力フォーマット

### テスト対象範囲
例:
- editor.html の保存導線
- builder.html のファイル検証から ZIP 生成まで

### ケース一覧
各ケースについて以下を記載する。

- ID
- 観点
- 確認内容
- 期待結果
- 優先度（High / Medium / Low）

例:

- ID: TC-01
- 観点: 入力・検証確認
- 確認内容: 必須ファイル未選択の状態で Build ZIP を実行する
- 期待結果: 実行が止まり、不足ファイルと対処方法が表示される
- 優先度: High

### 回帰確認項目
- 今回の修正で影響を受けそうな箇所を列挙する

### 未確認リスク
- 未実装
- 外部依存未接続
- 手動確認未実施
- モバイル未確認

## 判断ルール
- まず High の致命系から優先する
- 主要導線が最後まで通ることを重視する
- UI 崩れだけでなく利用者の詰まりも確認対象にする
- 実装範囲が限定的でも、影響範囲に応じて回帰項目を足す

## 禁止事項
- 抽象的な「動作確認する」で終わること
- 期待結果のないテストケースを作ること
- エラー系確認を省くこと
- モバイル確認を恒常的に後回しにすること

## 完了条件
- 実施すべきテスト観点が整理されている
- 優先順位付きのケース一覧がある
- 最低限の回帰確認項目が分かる
- レビュー前またはリリース前の確認に使える状態になっている
