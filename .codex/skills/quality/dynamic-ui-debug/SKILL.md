# dynamic-ui-debug

## 目的
動的 UI で
- 係数だけを無限に調整する
- 物理っぽさと見た目補正が混ざる
- 静止画 1 枚で pass 判定する

といった進め方を避け、切り分け順と確認観点をそろえる。

## 使う場面
- ドラッグ追従が不自然な UI
- 解放後の戻り方や減衰が不自然な UI
- 接続位置や追従関係が崩れやすい UI
- 座標変換、回転、クランプ、補正が絡む UI

## 最初に決めること
実装前に次を短く明文化する。

1. どこが基準点か
2. どの要素が主役で、どの要素が従属か
3. どこまでがロジックで、どこからが描画補正か
4. 完了判定で確認する状態は何か

## 切り分け順

### 1. 静止状態
- 触っていない時に、どこへ落ち着くべきかを先に決める
- この時点では操作感より「止まる向き」が正しいかを優先する

### 2. 操作追従
- 右へ動かしたら右、上へ動かしたら上など、期待方向と一致するかを見る
- 絶対位置より相対移動量で扱うほうが安定するかを確認する
- 入力経路が複数あるなら別々に切り分ける

### 3. 解放直後
- 離した瞬間に止まりすぎていないか
- 逆に勢いを持ち越しすぎて破綻していないか
- クランプや補正がここで不自然に効いていないかを見る

### 4. 減衰完了
- 中心付近で即停止していないか
- 逆に収束しなさすぎないか
- `damping` だけでなく、`stiffness`、速度持ち越し、速度上限もまとめて見る

### 5. 従属要素の見え方
- 主要要素とは別に、接続や追従が切れて見えないか
- 接続位置の正しさと、見た目の自然さを別問題として扱う

## 原則
- 止まる向きが間違っている時は、まず基準点と平衡条件を疑う
- 操作方向が逆な時は、まず座標変換と符号を疑う
- 戻らない時は、減衰より先に戻り先そのものを疑う
- 見た目だけ不自然な時は、ロジックを壊さず描画補正で解けるかを検討する

## 完了前の最低確認
- 静止状態
- 操作追従
- 解放直後
- 減衰完了
- 従属要素の見え方
- `git diff` に要求外の変更や文字化けがないこと

## Playwright MCP が不安定な時
- 合成操作が信用できない時は、無理に pass 判定しない
- 表示、スクリーンショット、DOM、座標計測までは Playwright で進める
- 手触りそのものは実操作確認を正とする
- 自動確認の限界を完了報告に残す

## 禁止事項
- ロジックの問題を見た目補正だけで隠して close すること
- 描画補正を入れたのに意図を残さないこと
- 静止画 1 枚だけで動的 UI を pass 判定すること
