# sprint-planning

## 目的
既存バックログから、次のスプリントで実施する範囲を決め、  
**スプリントゴール、対象タスク、依存関係、リスク** を整理する。

## 使う場面
- スプリント開始前
- 着手候補が多すぎて優先度を絞りたいとき
- MVP 達成までの短期計画を作りたいとき

## 入力
- `docs/05_TASKS.json` または backlog 一覧
- 現在の進捗状況
- Blocked 項目
- 依存関係や外部確認待ち
- 必要に応じて `docs/40_SPRINT_PLAN.md` の既存内容

## 出力
以下を含むスプリント計画案を出力する。

1. スプリントゴール
2. 対象タスク一覧
3. 今回入れない項目
4. リスク
5. 依存関係
6. 完了条件

## 出力フォーマット

### スプリントゴール
例:
- 編集者が WebAR 設定を最後まで完了できる MVP 導線を成立させる

### 対象タスク
各項目について以下を整理する。

- ID
- タイトル
- 目的
- 優先度
- 依存関係
- 完了条件
- 備考

### 今回入れない項目
- 後回しにする理由
- Blocked 理由
- 先に必要な判断

### リスク
- 仕様未確定
- 外部依存
- テスト未整備
- 実装量超過の懸念

## 判断ルール
- スプリントゴールは 1 つに絞る
- タスクはレビュー可能な粒度に分割する
- 依存関係が未解消のものは安易に入れない
- 実装だけでなく docs 更新や確認作業も含めて計画する
- 人間判断が必要な項目は先に見える化する

## 禁止事項
- ゴール不在のタスク寄せ集めにしない
- 未解消の依存を隠したまま詰め込まない
- 完了条件のない項目を混ぜない

## 完了条件
- スプリントゴールが明確
- 対象、非対象、リスクが整理されている
- 着手順を判断できる状態になっている
