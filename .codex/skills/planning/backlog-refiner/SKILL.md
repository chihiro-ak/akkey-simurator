# backlog-refiner

## 目的
企画書、要件、ユーザーストーリー、既存実装状況をもとに、  
Webアプリ開発向けの **プロダクトバックログ候補** を整理する。  
大きすぎる項目はエピック化し、必要に応じてタスク粒度まで分解する。

## 使う場面
- 初回バックログ作成時
- 仕様変更後のバックログ整理時
- タスクが大きすぎて着手しづらいとき
- MVP に必要な作業だけ先に見たいとき

## 入力
- `docs/01_PLAN.md`
- `docs/02_PRODUCT_CONTEXT.md`
- `docs/03_USER_STORIES.json`
- `docs/04_SPIKES.json`
- 既存の issue / backlog / メモ
- 実装済みファイルや README

入力が不足している場合でも、存在する資料だけで可能な範囲で整理する。  
不足情報がある場合は「不足情報」として明記する。

## 出力
以下を含むバックログ整理結果を出力する。

1. エピック一覧
2. バックログ項目一覧
3. 優先度
4. 依存関係
5. Definition of Done 案
6. 必要に応じて人間側タスク

## 出力フォーマット

### 1. エピック一覧
- EPIC-01: 例）主要導線の MVP 実装
- EPIC-02: 例）入力検証とエラー体験改善
- EPIC-03: 例）公開準備と運用手順整備

### 2. バックログ項目
各項目について以下を整理する。

- ID
- タイトル
- 種別（product / design / implementation / test / docs / ops / qa）
- 概要
- 優先度（High / Medium / Low）
- 依存関係
- DoD
- 備考

例:

- ID: BL-01
- タイトル: 編集者が scene-config を安全に作成できるように入力検証を追加する
- 種別: implementation
- 概要: 必須項目不足や参照切れを保存前に検出できるようにする
- 優先度: High
- 依存関係: なし
- DoD:
  - 必須入力チェックが定義されている
  - UI 上でエラー理由が分かる
  - 手動確認項目が更新されている
  - 関連 docs の更新対象が整理されている
- 備考: 仕様未確定なら仮置き前提を明記

### 3. 補足
- 不足資料
- リスク
- 先に決めるべきこと
- 外部確認待ち項目

## 判断ルール
- 大きすぎる項目はエピックにまとめ、実行可能な粒度に分割する
- 1項目1成果物を意識する
- Webアプリ開発では、実装だけでなく **要件、UI、テスト、運用、ドキュメント** もバックログ対象にする
- MVP の価値に直結する項目を優先する
- 外部依存や人間確認待ちは別タスクとして明示する
- 明確な判断材料がない場合は、仮説を置いて整理し、その前提を明記する

## 特に重視する観点
- 主要ユーザーフローの完成に必要な流れが抜けていないか
- 実装前提の不明点がタスク化されているか
- テストや確認作業が後回しになっていないか
- docs 更新が必要な項目を見落としていないか
- リリースや運用準備が最後に押し込まれていないか

## 禁止事項
- 不足情報を理由に整理を止めない
- いきなり細かすぎる技術タスクだけに落とし込まない
- 外部依存の存在を隠す
- DoD のないタスクを量産しない

## 完了条件
- バックログ候補が優先度付きで整理されている
- 大項目と小項目が区別されている
- 依存関係と不足情報が明記されている
- 着手候補が分かる状態になっている
