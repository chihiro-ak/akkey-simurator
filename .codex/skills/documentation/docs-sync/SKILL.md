# docs-sync

## 目的
実装、UI、API 連携、テスト、運用手順の変更内容に応じて、  
**更新すべきドキュメントを洗い出し、docs の記述差分や未反映事項を整える**。

## 使う場面
- タスク完了前
- PR 前
- 仕様変更後
- テスト結果を反映したいとき
- リリース準備前

## 入力
- 変更内容の要約
- 変更ファイル一覧
- `docs/` 配下の関連ファイル
- backlog / task 情報
- 必要に応じて QA やテスト結果

## 出力
以下を含む docs 更新整理結果を出力する。

1. 更新対象 docs 一覧
2. 更新理由
3. 追記、修正内容案
4. 未反映事項
5. 完了チェック

## 対象例
- `docs/02_PRODUCT_CONTEXT.md`
- `docs/03_USER_STORIES.json`
- `docs/22_UI_SPEC.md`
- `docs/32_API_CONTRACT.md`
- `docs/50_TEST_PLAN.md`
- `docs/60_RELEASE_CHECKLIST.md`

## 出力フォーマット

### 更新対象 docs
各ファイルについて以下を記載する。
- ファイル名
- 更新理由
- 更新内容案
- 優先度

例:
- ファイル名: `docs/22_UI_SPEC.md`
- 更新理由: 保存前バリデーションとエラー表示の仕様が追加されたため
- 更新内容案:
  - 必須項目エラー時の表示ルールを追記
  - 保存ボタンの非活性条件を追記
- 優先度: High

### 未反映事項
- 実装済みだが docs 未更新
- 仮決めのまま残っているもの
- 人間確認待ちのもの

### 完了チェック
- 仕様差分反映済み
- UI / API 差分反映済み
- テスト観点反映済み
- 運用手順反映済み
- リリースノート必要有無確認済み

## 判断ルール
- 変更内容に直接関係する docs を優先して洗い出す
- 実装変更だけでなく、要件、UI、テスト、運用差分も見る
- 「更新不要」の場合も、その理由を短く明記する
- 仮決め事項は仮と分かるように書く
- docs 間で矛盾がある場合は、どれを正本として合わせるべきか示す

## 特に重視する観点
- 実装と docs がズレていないか
- UI 仕様や API 条件が古いまま残っていないか
- テスト観点が現状に合っているか
- リリース、運用関連の反映漏れがないか

## 禁止事項
- 「必要なら更新」で終わること
- どの docs を直すべきか曖昧なままにすること
- 差分理由を書かずにファイル名だけ並べること
- 実装済み差分を未反映のまま完了扱いにすること

## 完了条件
- 更新対象 docs が明確
- 更新理由と内容案がある
- 未反映事項が見える
- タスク完了前チェックとして使える状態になっている
