# webapp-implement

## 目的
画面仕様、要件、既存コードをもとに、  
**Webアプリでの実装方針を整理し、実装時の抜け漏れや破綻を防ぐ**。

## 使う場面
- 新しい画面や導線を実装するとき
- フォームや入力検証を追加するとき
- API 連携やファイル処理を追加するとき
- 実装前に設計を軽く整理したいとき

## 入力
- `docs/13_REQUIREMENTS.md`
- `docs/21_SCREEN_FLOW.md`
- `docs/22_UI_SPEC.md`
- `docs/32_API_CONTRACT.md`
- 必要に応じて既存の HTML / CSS / JS / TS ファイル
- エラー要件
- 外部サービスや環境変数の情報

## 出力
以下を含む実装整理結果を出力する。

1. 実装対象の要約
2. 変更対象ファイル
3. データフローや状態整理
4. UI / API / バリデーション整理
5. 実装時の注意点
6. 必要に応じてコード方針やサンプル

## 出力フォーマット

### 実装対象
- どの画面や導線を実装するか
- ユーザーが何をできるようになるか
- 既存機能へどう接続するか

### 変更対象ファイル
- 対象ファイル
- 変更理由
- 影響範囲

### 状態整理
- 入力値
- UI 状態
- 非同期処理状態
- エラー状態

### UI / API / バリデーション整理
- 必須入力
- 検証ルール
- 外部連携条件
- 失敗時の表示

### 注意点
- 既存導線への影響
- モバイル表示への影響
- 参照ファイル切れの有無
- パフォーマンスや保守性の懸念

## 判断ルール
- 実装前に対象画面、入力、状態、エラーを構造化する
- バリデーションとエラーハンドリングを先に明示する
- 外部依存は明記し、未確定は未確定として扱う
- 既存導線を壊さないことを優先する
- 必要に応じて確認手順やテスト観点まで落とし込む

## 特に重視する観点
- 条件漏れがないか
- 状態管理が破綻しないか
- エラー時に利用者が詰まらないか
- 後から見て追いやすい実装方針になっているか
- 仕様と実装がズレにくいか

## 禁止事項
- 入出力や状態の整理をせずに実装を始めること
- 外部依存や失敗時の挙動を曖昧にしたまま進めること
- 既存導線への影響確認を省くこと

## 完了条件
- 実装対象が構造化されている
- 変更対象、状態、注意点が整理されている
- 実装着手しやすい状態になっている
