# FEAT-001 editor shell and flow

## タイトル
ユーザーが 1 画面でシミュレーション作成の流れを理解できる

## 背景
- MVP は 1 画面または少数画面で主要導線を完了できる構成を優先する
- runnable UI は、操作列とプレビュー列を持つ単一画面の方向性を採用している
- 後続 feature の実装基盤として、画面構成と状態表示位置を先に固定する必要がある

## 目的
- 初見ユーザーが「画像を入れる → 金具を選ぶ → 穴位置を調整する → プレビューを見る」の順序を迷わず辿れるようにする

## scope
- シミュレーション編集画面の全体レイアウト
- mobile / desktop のレスポンシブ再配置
- ステップ順序が分かる見出し、補助文言、操作エリアの整理
- empty / in-progress / ready / error を表示する状態エリアの設計
- 後続 feature が差し込めるデータ配置と UI スロットの整理

## out of scope
- 実際の画像バリデーション詳細
- 金具種類の最終確定
- 穴位置調整ロジック
- 揺れアニメーションそのもの

## done criteria
- 主要操作エリアが 1 画面で確認できる
- mobile 幅では縦積み、desktop 幅では操作エリアとプレビューが自然に再配置される
- どの操作を先に行うかが見出しや補助文言で理解できる
- empty / error / ready 状態の表示位置と表示条件が定義されている
- FEAT-002 以降が後付けでなく自然に接続できる構成になっている

## 検証方法
- `npm run dev` でアプリを起動し、狭い幅、中間幅、広い幅でレイアウト再配置を目視確認する
- Playwright E2E で初期表示、主要セクションの表示、表示順序、モード切替の基本導線を確認する
- 既存の画像アップロード、パーツ選択、プレビュー表示が壊れていないことをスモーク確認する

## feature completion criteria
- 単体の検証が上記方法で完了している
- `npm run dev` でアプリが立ち上がる
- 既存導線に影響がないことを確認できている
- ユーザーの手元でレビュー、動作確認が完了している

## UI states
- empty: 画像未選択の初期状態
- in-progress: 入力途中でプレビュー待ちの状態
- ready: 必須入力が揃ってプレビュー確認できる状態
- error: 必須入力不足または入力エラーがある状態
