# 02_product_backlog.md

## 前提
- 参照資料: `docs/00_product.md`, `docs/01_product_concept.md`
- 正とする UI: `src/App.tsx`
- backlog は MVP を先に成立させる順番で整理する
- MVP 完了条件は「初見ユーザーが 1 画面中心で画像を入れ、金具と穴位置を調整し、完成イメージと揺れ方の傾向を確認できること」とする

## runnable UI 確認結果
- 主要導線は 1 画面に集約する方針で問題ない
- 既存 runnable UI には `画像アップロード`, `パーツ選択`, `編集/プレビュー切替`, `簡易プレビュー` がある
- 現状 UI は、mobile では縦積み、desktop では操作列 + プレビュー列の構成になっている
- MVP に必要だが未充足の要素は `穴位置調整`, `状態別 UI`, `簡易シミュレーション結果の説明`, `操作完了までの導線明示` である
- `公開する`, `保存`, `2連アクキー`, `高精度物理` は backlog 上の拡張候補として扱う
- FEAT-001 は、現行 runnable UI をそのまま採用して close とする
- FEAT-002 以降は、現行 runnable UI の大枠レイアウトと主要導線を維持したまま機能追加する

## Epic

### EPIC-01 シミュレーション作成体験を成立させる
- 1 画面中心でアップロードから確認まで完了できる操作導線を整える
- MVP の主価値である「制作前の検討支援」を最短で届ける

### EPIC-02 見た目と動きの検討を支援する
- 金具差分、穴位置差分、簡易な揺れ表現を比較できるようにする
- 高精度ではなくても判断材料になるプレビュー品質を確保する

### EPIC-03 MVP の利用障壁を下げる
- empty / error / success / disabled を明示し、初見でも迷いにくくする
- mobile / desktop の両方で主要操作が破綻しない状態にする

## 追加タスクメモ
- FEAT-004 に、編集モードで本体画像自体をドラッグして表示位置を調整できるタスクを追加する
- FEAT-005 に、編集モード / プレビューモードの両方で透過画像を四角い枠で囲わず、そのままの見た目で表示するタスクを追加する
- FEAT-005 に、プレビューモードでマウス / タッチ操作をきっかけに揺らせるインタラクションを追加する
- FEAT-005 の揺れ方は `https://github.com/cihiro-ak/akkey-sim` を参照元として扱う。ただし 2026-03-11 時点ではリポジトリ内容を取得できなかったため、実装着手時に参照可能か再確認する

## 非 MVP だが backlog に保持する候補

| ID | タイトル | 優先度 | 理由 |
| --- | --- | --- | --- |
| CAND-001 | 2連アクキー対応 | Medium | ニーズはあるが穴位置制御とプレビュー難度が大きく上がるため |
| CAND-002 | 結果の共有 URL / 埋め込み | Medium | UI イメージには存在するが、MVP の主価値成立に必須ではないため |
| CAND-003 | 保存機能 | Medium | 認証やデータ設計が必要で、初期検証より先に入れる価値が低いため |
| CAND-004 | 高精度な物理シミュレーション | Low | MVP は「傾向確認」で十分であり、精度向上は後続検証に回すため |
| CAND-005 | 入稿データ自動生成 | Low | 非 goal に含まれており、MVP の学習コストも上げるため |

## 優先順位の判断基準
- ユーザーが 5 分以内に 1 回シミュレーション完了できる導線を先に作る
- runnable UI と整合する実装を優先し、新規体験の追加は後回しにする
- 見た目確認の価値に直結する要素を先に実装し、保存や共有など周辺機能は後続に回す
- 依存関係が強い要素はシェルから順に小さく閉じる

## 共通完了条件
- feature 単体の検証が完了している
- 検証方法が明記されている
- UI 操作が対象の feature は Playwright による E2E テスト、または同等のブラウザ操作テストで確認されている
- `npm run dev` でアプリが正常に起動し、対象 feature が実画面で確認できる
- 既存の主要導線に影響がないことをスモーク確認できている
- `typecheck` と `build` が通る
- ユーザーの手元でレビューと動作確認が完了している
- 未確定事項や既知の制約があれば feature spec または実装メモに残っている

## 実装順序
1. FEAT-001 は現行 runnable UI を確定版として close 済み
2. FEAT-002 と FEAT-003 で入力値を安定して作れるようにする
3. FEAT-004 で MVP の差別化要素である穴位置検討を成立させる
4. FEAT-005 で最終確認体験を閉じ、empty / error / success を横断確認する

## 実装方針メモ
- FEAT-002 以降は、`src/App.tsx` の runnable UI をベースラインとして扱う
- ユーザー合意なしに、主要ブロック構成、主要導線、見た目の印象を大きく変えない
