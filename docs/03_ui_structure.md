# 03_ui_structure.md

## 前提
`docs/02_product_backlog.md` と `docs/00_ui_image.png` を読み直した結果、MVP の UI は **2 画面構成** で考えるのが自然。

- 画面 1: シミュレーション設定画面
- 画面 2: プレビュー画面

前回のように「mock を見せるための切替 UI」を持つのではなく、ユーザーが実際に触る導線として整理する。

## なぜ 2 画面構成か
ラフ画像では、左に `1.画像選択` `2.金具選択` `3.位置調整` `シミュレーションする` がまとまっている。  
右は `プレビュー専用画面` として分かれており、`戻る` と `公開する` が置かれている。

これは backlog の以下とも一致する。

- FEAT-001: 画像アップロードと初期設定
- FEAT-002: 金具選択と穴位置調整
- FEAT-003: 完成イメージのプレビュー

つまり、MVP では次の流れを UI にそのまま落とすべき。

1. 1 画面目で必要情報をすべて入力する
2. `シミュレーションする` で 2 画面目へ進む
3. プレビュー確認後に `戻る` で再調整する

## 画面 1: シミュレーション設定画面

### 役割
画像、金具、穴位置を一気に設定する画面。  
「順番に埋めていけば最後にシミュレーションできる」ことが重要。

### レイアウト
- 上部
  - タイトル `アクキーシミュレーター`
- 本文
  - 1. キーホルダーにしたいイラストを決める
  - 2. 金具を選ぶ
  - 3. アクリルと金具の穴位置をそろえる
- 下部
  - `シミュレーションする` ボタン

### UI 構成
#### 1. 画像選択ブロック
- 見出し
  - `1. キーホルダーにしたいイラストを決める`
- 左または上に小さな画像プレビュー
- 右または下に `画像選択` ボタン
- 画像選択後は小プレビューが更新される

#### 2. 金具選択ブロック
- 見出し
  - `2. 金具を選ぶ`
- 金具カードを横並び、モバイルでは縦並び
- MVP では 2 種から開始
  - ナスカン
  - ストラップ

#### 3. 位置調整ブロック
- 見出し
  - `3. アクリルと金具の穴位置をそろえる`
- 大きめの調整エリアを置く
- 中に
  - アクリル本体
  - 金具
  - 接続位置
- 基本操作はドラッグ
- ただし backlog のリスクどおり、ドラッグだけだと難しい可能性があるため補助 UI を追加する
  - `中央に戻す`
  - 微調整用の矢印ボタンまたは簡易スライダー

### CTA
- 画面最下部に `シミュレーションする`
- 必須条件未達なら disabled
- 有効化条件
  - 画像がある
  - 金具が選択されている
  - 接続位置が成立している

## 画面 2: プレビュー画面

### 役割
設定結果を確認する専用画面。  
ここでは「編集」より「確認」を主役にする。

### レイアウト
- 上部
  - タイトル `アクキーシミュレーター`
- 中央
  - 大きなプレビュー領域
- 下部
  - `戻る`
  - `公開する`

### UI 構成
#### プレビュー領域
- できるだけ画面の主役として大きく取る
- 金具付きアクキーを中央表示
- MVP では軽量な簡易プレビューでよい
- 揺れ確認は
  - 自動で軽く動く
  - またはボタンで再生する

#### 下部 CTA
- `戻る`
  - 設定画面に戻る
- `公開する`
  - MVP では本実装しない可能性が高い
  - その場合は以下のどちらか
    - 非表示
    - `準備中` 表示付き disabled

## 状態設計

### 設定画面
#### empty
- 画像未選択
- 金具未選択
- 調整エリアはプレースホルダ
- `シミュレーションする` は disabled

#### error
- 非対応画像
- 読込失敗
- 金具未選択
- 位置が不正

#### success
- 必須条件がそろい、`シミュレーションする` が押せる

### プレビュー画面
#### loading
- プレビュー生成中表示

#### error
- 必要データ不足
- 描画失敗
- `戻る` を主導線にする

#### success
- 完成イメージ表示
- 再確認または再調整ができる

## モバイル構成
### 設定画面
- 3 ブロックを縦積み
- 画像選択
- 金具選択
- 位置調整
- 最下部に `シミュレーションする`

### プレビュー画面
- 上から
  - タイトル
  - プレビュー
  - `戻る`
  - `公開する`

## 今回の UI 判断
- 画面は 1 枚完結ではなく、**設定画面とプレビュー画面の 2 画面** にする
- 設定画面内では、ステップタブ切替ではなく **縦に順番を見せる**
- プレビュー画面は説明要素を増やさず、プレビューを主役にする
- `公開する` は backlog どおり MVP 必須にはしない

## 次に mock へ落とすときのルール
- UI mock はこの 2 画面構成から外れないこと
- 画面確認用の余計な切替 UI は入れないこと
- FEAT-001 と FEAT-002 は設定画面に統合すること
- FEAT-003 は独立画面として扱うこと
